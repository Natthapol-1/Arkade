# Ruby Star — Agent Notes

A top-down, chamber-based space-station defense game. Four chambers (ALPHA/BETA/GAMMA/DELTA) in a 2×2 grid, linked by hallways, each with a teleport pad at its center. Guard the Ruby Core — if either it or the player hits 0 HP, it's game over. 12 enemy types, 5 boss encounters (mutually exclusive), a meteorite hazard, and 5 player abilities.

## Files

- **`engine.ts`** — All core game logic and state. Contains the `GameState`/`Enemy` interfaces, the main `tick()` loop, per-enemy-type update dispatch (`updateEnemies`), all ability handlers (`useLaser`, `useBomb`, `activateWave`, `useSpeedBoost`, `useBullet`, `toggleCarryRuby`, `tryActivateTeleport`/`doTeleport`/`cancelTeleport`, `healRuby`), the 5-way boss spawn timer/`godSpawnBoss`, and the meteorite system. Operates independently of rendering — no DOM/canvas references.
- **`constants.ts`** — Map generator (`generateMap`), chamber bounds/spawn tiles, hallway bounds (`HALLWAY_BOUNDS`/`hallwayOfTile`), tile types, `EnemyType` union + `ENEMY_CONFIGS` (hp/speed/color/damage/cooldown per type), difficulty tiers, and every tuning constant for abilities and all five bosses.
- **`page.tsx`** — The React component: `requestAnimationFrame` game loop, keyboard/mouse/touch input, the `drawGame`/`drawEnemy`/`drawMinimap` canvas rendering, `RulesModal` content, HUD, and the mobile-only virtual joystick + ability buttons (`TouchAbilityButton`).

## State shape (`engine.ts`)

- `GameState`: `gamePhase` (`'playing' | 'teleporting' | 'lost'`), player position/stats, ruby position/HP (`rubyTileX/Y === -1` means it's being carried), `enemies: Enemy[]`, per-ability cooldowns, `starEnergy`/`poweredTicks`, meteorite timers, boss timers (`bossTimer`, `bossWarningTicks`, `bossesKilled`), all visual-FX arrays (`laserBeams`, `waveEffects`, `bombBlasts`, `lightningArcs`, `deathParticles`), and teleport state (`teleportDestOptions`, `teleportCooldown`).
- `Enemy`: shared fields (`tileX/Y`, `targetTileX/Y`, `stepProgress` for tile-to-tile interpolation, `hp`, `attackTimer`, `windupTicks`, `shootTicks`, `flashTicks`, `healFlashTicks`). Several fields are dual-purpose per type — see gotchas below.

## Non-obvious behavior

- **Teleportation pauses the whole game.** `tick()` returns immediately (`if (state.gamePhase === 'teleporting') return;`) — enemies, cooldown timers, the meteorite clock, and the boss spawn timer all freeze while the destination menu is open. Pressing `Spacebar` on a pad opens it; picking a chamber or pressing Escape resumes normal ticking. If Space is pressed while *not* on a pad, it falls back to `healRuby`.
- **Bomb hurts the player, never the Ruby.** `useBomb`'s detonation damages any enemy in blast radius, damages the player if they're still in range (35% of the bomb's damage), but has no `damageRuby` call at all — the Ruby Core is immune to your own bomb.
- **Power-up window, not a charge.** Reaching max `starEnergy` immediately consumes it and grants `poweredTicks = 120` (~2s). *Any* ability fired during that window is enhanced (more range/damage/duration) — it's a timed buff, not a single-use token.
- **Fiery King vs. Shadow Queen vs. Storm Reaper vs. Soul Devourer vs. Frost Warden are mutually exclusive.** The spawn timer (`BOSS_SPAWN_INTERVAL`, ~12s after the previous one dies) checks all five types before allowing a new warning — only one boss-tier enemy exists at a time, and which type spawns is a random pick of the five made at spawn time (the warning banner stays generic "BOSS APPROACHING" either way, so the type is a surprise). The very *first* boss of a run is deliberately delayed further than subsequent ones — `createInitialState` seeds `bossTimer` with `BOSS_FIRST_SPAWN_TIMER` instead of `BOSS_SPAWN_INTERVAL`, sized so the first spawn lands around 60s in (accounting for the `difficultyLevel >= 1` gate at ~10s and the warning banner's ~3.5s), while every boss after that still uses the shorter `BOSS_SPAWN_INTERVAL`. A god-mode-only debug tool (`godSpawnBoss`, gated behind `?god=1`) can force-spawn any of the five on demand, despawning whichever boss currently exists (queen decoys are untouched; an active hallway seal or Frost Warden ice-tile set is also lifted immediately).
- **Frost Warden (`frost_warden`) is a stationary field-controller with zero direct offense.** She never chases and never attacks the player or ruby (`damageToPlayer`/`damageToRuby` are 0 and unused). Her threat is entirely passive: while the player shares her chamber (`isPlayerChilled`, checked fresh every tick, not stored), player move speed is multiplied by `FROST_CHILL_SPEED_MULT` (0.65) and the LASER/BULLET/WAVE cooldowns (and the WAVE hold-to-charge duration in `page.tsx`) tick down at `FROST_CHILL_TICK_RATE` (0.5x — effectively half attack speed) instead of the normal 1/tick. On spawn and every `FROST_RELOCATE_INTERVAL` (~30s, reuses `phaseTimer` like the Queen's phase-jump) she teleports to a different chamber and calls `scatterFrostIceTiles` to pick `FROST_ICE_TILE_COUNT` (7) truly random floor tiles from that chamber's interior (scanning `CHAMBER_BOUNDS` for `T_FLOOR`, not the perimeter-only `CHAMBER_SPAWN_TILES` pool that was used at first) as hazards (`state.frostIceTiles`, not a tile-type mutation — the underlying tile stays `T_FLOOR`, hazard membership is just a coordinate-list lookup). Stepping onto one of these tiles sets `state.playerFrozenTicks = FROST_FREEZE_DURATION` (~1.5s), which makes `updatePlayer` return immediately at the top — full immobilization, not just a slow — and plays a thaw sound the instant it expires. Her only "offense" is supportive: every `FROST_SHIELD_CAST_DELAY` (~10s, recurring, reuses `windupTicks`) she fires a visible ice-laser (`castFrostShield`) at one random non-boss ally sharing her chamber, granting `FROST_SHIELD_HP` (10) points of `icyShieldHP` — a real absorb pool (`damageEnemy` drains it before touching `hp`), not a %-reduction like the shielder's. Any number of allies can be carrying a shield at once, since it's a per-`Enemy` field, not a single global slot. Dying (or being god-mode-despawned) clears `frostIceTiles` immediately.
- **Soul Devourer (`devourer`) grows mid-fight by eating other enemies.** Targets whichever of player/ruby is closer, like the King, and attacks at range with a "sound wave" burst (`state.laserBeams`, color `'#6b2fa8'`, rendered with a dedicated wavy/rippling `isDevourerLaser` style in `page.tsx` rather than a solid bolt) instead of a plain melee bite — her `attackRange` grows +1 tile every 5 absorb stacks, so by the time she's stacked up she's shouting from well outside melee. Every `DEVOURER_ABSORB_COOLDOWN` (~9s — deliberately slow so reaching all 20 stacks takes ~3 minutes minimum even with a constant supply of nearby grunts), she finds the nearest non-boss ally within `DEVOURER_ABSORB_RANGE` and consumes it outright (removed from `state.enemies`, no kill reward for the player). Below `DEVOURER_MAX_STACKS` (20) this *compounds* her `maxHp` by `DEVOURER_STACK_HP_PCT` (12%, and heals her by the same amount gained), scales her damage on an accelerating quadratic curve (`DEVOURER_STACK_DMG_MAX_MULT`, base 12 dmg → 90 at full stacks — small early jumps, big ones late) per stack, and slows her by `DEVOURER_STACK_SPEED_PCT` (4%, linear, floored at 0.15 speed) per stack. **She never stops eating once maxed out** — at 20 stacks, absorbing an ally no longer grows her further, it just tops her `hp` back to `maxHp` (a free heal-to-full), so parking nearby grunts near a fully-stacked Devourer to bait a heal doesn't actually make her stronger. The compounding HP growth is deliberate — at 20 stacks she's roughly 10x her base HP (576) and 7.5x her base damage (90), "insanely broken" if a fight is neglected long enough to let her feast unchecked, but harmless if the grunts near her are cleared or she's rushed early. Two fields are reused for her: `chargeDirX` (unused otherwise for this type) holds the absorb-stack count, and `windupTicks` (also unused otherwise) holds the absorb cooldown countdown.
- **Storm Reaper (`storm_reaper`) always hunts the player, never the ruby**, unlike the King and Queen (both of which target whichever of player/ruby is closer/present) — a deliberate "assassin" identity (`damageToRuby: 0`). She BFS-chases at a flat speed (no ramp, unlike the King) and melee-strikes at range 1. If she and the player are both standing in the same hallway (`hallwayOfTile`), and neither a seal is already active nor its cooldown (`REAPER_SEAL_COOLDOWN`) is up, she seals that hallway — `setHallwayWalled` walls off only the two end-caps of its `HALLWAY_BOUNDS` rect (not the whole corridor — sealing every tile stranded whoever was inside with zero adjacent floor tiles, freezing them in place instead of trapping them in a fightable space) — for `REAPER_SEAL_DURATION` (~4s), then reopens automatically. Since hallways are pure floor by construction (no cover-walls or teleport pads inside them, confirmed against `generateMap`), the wall swap is always clean and reversible. The seal state lives on `GameState` (`hallwaySeal`/`hallwaySealTicks`/`hallwaySealCooldown`), not per-enemy, since only one Reaper can ever exist. Dying (or being god-mode-despawned) immediately lifts an active seal rather than waiting out the timer.
- **Shadow Queen never chases.** She stays put in her chamber, firing through walls (no line-of-sight check, just same-chamber + `QUEEN_ATTACK_RANGE` distance) at whichever of player/ruby is present. Every `QUEEN_PHASE_INTERVAL` (~8s) she teleports to a different chamber — excluding both her current chamber *and* whichever chamber the player is currently in — leaving a `queen_echo` behind at her old tile.
- **Fiery King targets whichever of player/ruby is closer** (mirrors the Queen's split-attention behavior, computed fresh each tick via `tileEuclidDist` rather than gated by chamber like hers) — he'll BFS straight for an undefended ruby if it's nearer than the player. Only the Reaper is exempt from this and stays player-exclusive.
- **Echoes persist until killed.** `queen_echo` has 1 HP, keeps firing on the same windup/range rules as the Queen, and does *not* expire on a timer — `phaseTimer` is unused for this type (only meaningful for `splitter_queen`, where it's the phase-jump countdown). Ignoring the Queen's jumps lets echoes stack indefinitely. Killing the *real* Queen dissolves every remaining echo instantly (no reward for that cleanup — `killEnemy`'s `skipRewards` flag covers both meteor kills and echo kills).
- **Meteor strikes cycle chambers clockwise, not randomly.** Order is ALPHA(TL) → BETA(TR) → DELTA(BR) → GAMMA(BL) → repeat, tracked by `meteoriteChamberSeq` indexing into `METEOR_CLOCKWISE_ORDER`. All five boss types take `BOSS_METEOR_DMG`% of their *current* max HP (not insta-killed like every other enemy type in the struck chamber) — for the Soul Devourer this uses her post-absorb, grown `maxHp`, so a heavily-stacked one still takes a meaningful hit. For the King/Queen/Reaper/Frost Warden, anything already at or below that HP fraction dies outright; the **Soul Devourer is the one exception** — a meteor alone can never kill her, her HP is floored at 5% of her current `maxHp` instead of being allowed to hit 0. `queen_echo` has no special case and dies outright like normal enemies. Note meteor strikes only target `CHAMBER_BOUNDS`, not hallways, so a Reaper lurking mid-hallway is untouched by them.
- **Ruby Carrying:** picking up the Ruby (E/F) halves player movement speed but makes enemies target *only* the player; placing it lets enemies split attention and attack the Ruby directly.
- **Death particles vs. FX arrays.** Every enemy death spawns a small colored particle burst (`spawnDeathParticles`, reuses the enemy's own config color) via `state.deathParticles` — separate from the beam/wave/blast FX arrays, and separate again from `spawnFireBurst` (used specifically for the Fiery King's laser landing on the player, orange/yellow embers with a slight upward kick).
- **King vs. Queen vs. Reaper vs. Soul Devourer vs. Frost Warden are visually unrelated on purpose.** The Fiery King's ambient effects (rising embers, ground-fire glow, flickering flame-lick spike tips, fire-gradient laser with an ember-storm trail), the Shadow Queen's (orbiting motes, revolving segmented ring, twinkling gems, magic-shimmer laser with an arcane-ripple + dust-storm trail), the Storm Reaper's (rotated-diamond core, angular blade fins, orbiting zigzag electric arcs, motion-streak afterimages, no crown/tiara since she's a rogue not royalty), the Soul Devourer's (violet/green corruption palette, body that physically grows and sprouts fused chunk-blocks with each absorb stack, single glowing eye, blocky maw that widens when attacking, always-on aura ring + orbiting spore particles), and the Frost Warden's (icy blue-white palette, crystalline 3-spike icicle crown instead of a gold crown or tiara, drifting frost-mote orbit, wavy rippling "sound wave"-style frost bolt) are each built from different primitives so none read as a reskin of the others. All five still use the same blocky, rectangle-based construction style as the rest of the game's sprites (`fillRect`/`strokeRect`, not smooth curves) — an earlier organic/curved take on the Soul Devourer (tendrils, circular chunks) was reworked into blocks for consistency, and the Frost Warden was built blocky from the start. The `EnemyType` key is `'fiery_king'`; some internal identifiers (`isBossLaser`, `godSpawnBoss`, etc.) still say "boss" as a naming holdover and aren't tied to the type string.
- **Mobile controls are DOM siblings, not descendants, of the canvas's touch-handled wrapper div.** The virtual joystick and ability-button cluster in `page.tsx` sit in separate fixed-position elements after the canvas wrapper closes, so a touch starting on them never bubbles into the wrapper's own `onTouchStart/Move/End` (swipe-to-move) or `onMouseDown/Up` handlers. They're shown/hidden purely via CSS (`@media (hover: none) and (pointer: coarse)`), keyed off actual input capability rather than viewport width, so anything with a real mouse never renders or sees them. Every button calls the *same* engine function keyboard/mouse call (e.g. the LASER button reuses `mouseDownTimeRef` so hold-to-charge-WAVE behaves identically) — there is no parallel mobile-only game logic.
- **Canvas Rendering:** `drawEnemy` has fully custom, early-`return`ing draw branches for `fiery_king`, `splitter_queen`, `queen_echo`, `storm_reaper`, `devourer`, and `frost_warden` (each builds its own body/eyes from scratch); every other type shares one generic blocky-body-plus-per-type-antenna/eyes path at the bottom of the function.
