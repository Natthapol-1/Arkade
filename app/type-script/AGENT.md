# Type:Script — Agent Notes

A typing-defense game: words fall from the top of a field toward your
"Core" at the bottom. Type a word exactly and press Enter to destroy it
before it breaches. Fourteen word "kinds" layer different twists on top of
the same base loop, unlocking one at a time as you level up.

## Files

- **`constants.ts`** — types (`WordKind`, `FallingWord`, `GameState`, `SpeedEffect`), the word banks (`WORDS_SHORT`/`WORDS_MEDIUM`/`WORDS_LONG`, plus `CHAIN_PAIRS` and `WORDS_CASE_SENSITIVE`), per-kind display metadata (`KIND_STYLE` — accent color + rules-modal hint text, also used live in the HUD/rules), unlock levels (`KIND_UNLOCK_LEVEL`), all timing/scaling tunables, and `requiredInput(word)` — the single source of truth for "what string actually destroys this word" (see below).
- **`engine.ts`** — pure functions, no React/DOM: `spawnWord`/`spawnChainedPair`, `tick` (advance + breach + expire effects + mutate `shifting` text + auto-spawn), `submitInput` (resolve an Enter-confirmed line against on-screen words — including case-sensitive matching and `spawner` children), `chainPairs` (groups on-screen `chained` halves into ordered pairs — exported because both `submitInput` and `page.tsx`'s rendering need the same grouping), plus `spawnIntervalForLevel`/`fallSpeedForLevel`.
- **`page.tsx`** — game loop (`gameRef` + `setInterval(TICK_MS)`, same ref/state-split pattern as Painting Python — mutating `gameRef.current` doesn't re-render, so every tick also bumps `renderTick`), the controlled `<input>` that's the entire control scheme (also passed into `WordGlyphs` as a prop, since `vanish` needs to know live keystrokes to decide what to mask), and per-kind rendering.

## `requiredInput` — why it exists

Every kind except `skipLetter` requires typing the word's canonical `text` verbatim. `skipLetter` is the one exception: one character index (`skipIndex`, chosen at spawn) must be **omitted** from the typed input. `requiredInput(word)` centralizes this so `engine.ts`'s matching logic and any future kind-specific input rule never has to special-case it more than once. If you add a kind whose required input differs from its display text (the way `reversed` differs in *display* but not *required input* — typing the normal spelling still matches `word.text` directly), put that divergence here, not in `page.tsx`.

## The fourteen kinds

| Kind | Unlocks at | Twist |
|---|---|---|
| `normal` | always | Baseline — type it, Enter, destroyed. |
| `shielded` | Lv 2 | Needs **two** separate type+Enter passes. First correct submission sets `shieldBroken: true` (word survives, no score yet); second destroys it and scores. `requiredInput` is the same both times — `submitInput` branches on `shieldBroken` to decide hit vs. destroy. |
| `reversed` | Lv 3 | Displayed via `reversedText()` in `page.tsx` (cosmetic only). Required input is still the normal spelling — `requiredInput` does nothing special for this kind. |
| `erratic` | Lv 4 | Movement only. `tick` computes `x = baseX + sin(elapsed/450 + erraticSeed) * 10`, clamped to `[EDGE_MARGIN, 100 - EDGE_MARGIN]`. No typing difference from `normal`. |
| `virus` | Lv 5 | The only kind where **destroying it is the wrong move**. Typing it correctly (`requiredInput` matches) costs a life (`virusPenalty` outcome) and removes it. Left alone to breach, `tick` explicitly skips the life deduction (`if (w.kind !== 'virus') lives -= 1`) — ignoring it is free and correct. |
| `slow` | Lv 6 | On destroy, pushes a `{ kind: 'slow', expiresAt }` into `state.effects` — a temporary *global* multiplier on every word's fall speed (see below). |
| `boost` | Lv 7 | **Not** a global effect (it used to be — see git history if you need the old version). A boost word simply falls at `BOOST_WORD_MULTIPLIER` (1.6×) its own speed at all times, computed per-word inside `tick`'s loop (`speed = w.kind === 'boost' ? baseSpeed * BOOST_WORD_MULTIPLIER : baseSpeed`). Destroying it scores at a 1.2× multiplier (`scoreForWord`) to reward the faster reaction it demands — it doesn't touch any other word on screen. |
| `skipLetter` | Lv 8 | `skipIndex` is rolled at spawn (`spawnWord`); the player must type every letter **except** that index, in order. `page.tsx`'s `WordGlyphs` renders the marked letter in `var(--danger)` with a strikethrough so it's visually distinct from "just another letter." |
| `chained` | Lv 9 | The one kind that isn't a single `FallingWord` — see below. |
| `shifting` | Lv 10 | `tick` mutates `text` to a new random word once `now >= nextShiftAt`, resetting `nextShiftAt = now + SHIFT_INTERVAL_MS`. See below. |
| `spawner` | Lv 11 | Destroying it spawns `SPAWNER_CHILD_COUNT` 1-letter `normal` words at its last position. See below. |
| `vanish` | Lv 12 | Display only, in `page.tsx`. See below. |
| `encrypted` | Lv 13 | Display + accent only, in `page.tsx`/`constants.ts`. See below. |
| `caseSensitive` | Lv 14 | The one kind where matching is exact-case. See below. |

Kinds are otherwise orthogonal — nothing here stops a future change from, say, making an `erratic` word also `shielded`; today each spawn gets exactly one `kind` (`spawnWord` picks one via `pickKind`'s weighted random over currently-unlocked kinds). `chained` is the exception: picking it spawns *two* `FallingWord`s in one go (see below), so it can never combine with another kind by construction.

## `chained` — two words, one obstacle

Picking `chained` in `pickKind` routes to `spawnChainedPair` instead of the normal single-word path (`spawnWord`'s very first line: `if (kind === 'chained') return spawnChainedPair(state);`). It pulls one `[string, string]` tuple from `CHAIN_PAIRS` (real two-word collocations like `['BUG', 'DETECTED']`, deliberately chosen so the phrase reads naturally) and creates two `FallingWord`s, both `kind: 'chained'`, sharing a freshly-generated `chainId`. Everything else about them (`spawnedAt`, fall speed) is identical to a normal word, so **they always stay in perfect lockstep** — same `y` every tick, forever — there's no special sync logic needed because they're never treated differently from any other word by `tick`'s fall-speed math.

- **Lane order encodes reading order.** `pickLanePair` returns two lanes already sorted ascending, and the *first* element of the `CHAIN_PAIRS` tuple always goes to the lower lane. This means whichever word renders on the left is always the grammatically-first word — the player just reads left-to-right and types what they see, exactly like normal text, with a space in between. There's no separate "phrase order" field to keep in sync; `chainPairs` recovers reading order purely by sorting on `w.lane`.
- **Matching:** `submitInput` checks `chainPairs(state.words)` *before* falling through to the normal single-word `matches` filter (which explicitly excludes `kind === 'chained'` — `w.kind !== 'chained' && requiredInput(w) === input` — so half a pair can never be destroyed alone, even by accident). A match requires the exact string `` `${first.text} ${second.text}` ``; both halves are removed together, scored together (`(first.length + second.length) * 10 * CHAIN_SCORE_MULTIPLIER`), and count as **one** `destroyedThisLevel` increment, not two.
- **Breaching:** because both halves share identical `y` always, they breach in the exact same tick — `tick`'s life-loss loop dedupes by `chainId` (a `Set<string>` of chain-ids already charged this tick) so letting a pair fall through costs exactly **one** life for the pair, not one per word.
- **Rendering:** `chainPairs` (exported from `engine.ts`) is reused as-is by `page.tsx` to draw a `.breach-chain-link` div between the two words each render — a glowing bar from `first.x%` to `second.x%` at their shared `y%`, `z-index: 1` versus `.breach-word`'s `z-index: 2` so the link visually passes *behind* both word boxes (it's only visible in the gap between them).
- **Orphan safety:** `chainPairs` only emits a pair when a `chainId` group has exactly 2 members. Nothing in the current code can produce an orphaned single half (both halves are always added and removed together), but the guard is cheap insurance if that invariant ever breaks.

## `shifting` — text mutates mid-fall

`FallingWord.nextShiftAt` is set at spawn (`state.now + SHIFT_INTERVAL_MS`) and checked every tick in `tick`'s per-word loop: once `now >= nextShiftAt`, the word's `text` is replaced (`pickShiftText`, which retries up to 5 times to avoid picking the same word again) and `nextShiftAt` is pushed out another `SHIFT_INTERVAL_MS`. There's deliberately **no special handling for "the player was mid-typing when it shifted"** — the input box just holds whatever string the player has typed; if the word's `text` changes before they hit Enter, their finished input will no longer equal the new `text` and `submitInput` reports a plain `miss`. The "you have to finish before it shifts" rule the player experiences falls out naturally from there being nothing else propping up their now-stale progress.

## `spawner` — destroying it backfires into a swarm

Resolved entirely in `submitInput`'s normal destroy branch: `else if (target.kind === 'spawner') { words = [...words, ...spawnChildrenAt(target.x, target.y, state.now)]; }`. `spawnChildrenAt` creates `SPAWNER_CHILD_COUNT` (3) single-letter `kind: 'normal'` words, fanned out a little horizontally (`x ± 8%` per child, clamped to the same `EDGE_MARGIN` bounds everything else respects) and spawned **at the parent's last `y`**, not `y: 0` — they're already partway down the field, giving much less reaction time than a fresh spawn. They use `lane: -1` (not a real lane) since they aren't part of the normal spawn-interval flow and don't need overlap avoidance against future spawns; this is a harmless sentinel, never matches the `0..LANES-1` range `pickLane`/`pickLanePair` check against.

## `vanish` — masked the instant you start typing it

Purely a `page.tsx` rendering concern; the engine treats a `vanish` word exactly like `normal` (same `requiredInput`, same scoring path with a 1.4× multiplier in `scoreForWord`). `WordGlyphs` now takes an `inputValue` prop (passed down from the page's controlled `<input>` state) and masks to `'*'.repeat(word.text.length)` whenever `inputValue` is non-empty **and** is a prefix of `word.text` (both upper-cased for the comparison, consistent with every non-`caseSensitive` kind). The moment the player's input no longer matches that prefix (cleared, or typed wrong), it reverts to showing the real text again. If two `vanish` words on screen happen to share a prefix, both mask simultaneously — an accepted, rare edge case, not specifically resolved.

## `encrypted` — garbled until halfway down

`garbledText` (a same-length string of random symbols from `SYMBOL_CHARS`) is generated once at spawn (`spawnWord`) and never changes. `WordGlyphs` shows `garbledText` while `word.y < ENCRYPTED_REVEAL_Y` (50) and the real `word.text` once it crosses that line; `wordAccent` similarly dims the word to `var(--text-muted)` pre-reveal so it visually reads as "locked." **Matching is display-independent** — `requiredInput`/`submitInput` always compare against the real `word.text`, reveal state or not. This is intentionally not enforced as a hard block: there's no realistic way to type the correct word before seeing it, so the engine doesn't need extra bookkeeping to prevent an "early" correct guess.

## `caseSensitive` — exact spelling, including case

Every other kind matches case-insensitively: `submitInput` upper-cases the trimmed input once (`input = rawTrimmed.toUpperCase()`) and compares against `requiredInput(word)` (itself always upper-case, since every other word bank is authored in caps). `caseSensitive` words break that pattern on purpose — their bank (`WORDS_CASE_SENSITIVE`) is authored in literal mixed case (`'aCcEsS'`, `'snake_Case'`, ...) and stored as-is, never upper-cased anywhere. The match check special-cases it: `w.kind === 'caseSensitive' ? w.text === rawTrimmed : requiredInput(w) === input` — comparing against the **untouched** trimmed input, not the upper-cased one. Get this backwards (e.g. accidentally upper-casing `WORDS_CASE_SENSITIVE` itself, or comparing it against `input` instead of `rawTrimmed`) and the kind silently becomes case-insensitive like everything else, defeating the point.

## Speed effects (`slow`) vs. per-word speed (`boost`)

`state.effects: SpeedEffect[]` is a flat list of `{ kind: 'slow', expiresAt }` — `'slow'` is the only kind left in this union; `boost` was migrated off this mechanism (see above) and `SpeedEffect['kind']` would need widening again if a future kind wants a real global effect. `activeSpeedMultiplier` (in `engine.ts`) filters to unexpired entries and multiplies all active factors together (today there's only ever one factor possible, `SLOW_FACTOR`, but multiple simultaneous `slow` words refresh rather than stack — see `submitInput`'s `effects.filter(e => e.kind !== 'slow')` before pushing the new one). `tick` prunes expired effects every call; nothing else needs to know about expiry. `boost`'s per-word multiplier, by contrast, lives entirely inside `tick`'s per-word loop and never touches `state.effects` — it has no duration, no expiry, and no effect on any other word.

## Loss condition & lives

`LIVES_START = 10` (`constants.ts`). Every non-virus breach (`tick`) and every virus-mistyped submission (`submitInput`) deducts exactly one life regardless of word kind/length — there's no graded damage. `status` flips to `'lost'` the instant `lives <= 0`, checked in both `tick` and `submitInput` (a virus penalty can end the game on its own, so it can't only be checked in `tick`). Once lost, **`gameRef.current` is left as-is** (unlike Painting Python, which immediately resets its ref and has to snapshot the score first) — `page.tsx` reads `game?.score`/`game?.level` directly for the game-over popup, and only `handlePlayAgain` actually calls `createInitialState` again.

**Testing aid — `godMode`:** visiting `/type-script?god=1` sets `GameState.godMode = true` (read once via `readGodMode()` in `page.tsx`, passed into every `createInitialState` call — including `handlePlayAgain`, so it survives "PLAY AGAIN"). Both life-loss sites (`tick`'s breach loop, `submitInput`'s virus branch) are gated with `if (!state.godMode)`, and both `status: 'lost'` assignments are gated the same way. The HUD shows `∞` instead of the lives battery when active (`page.tsx`, the Lives `hud-value`/`.battery`). There's no UI toggle for `?god=1` itself on purpose — it's a URL-param cheat, not a player-facing feature — but once it's on, a "⏩ LV+5" button appears next to "? RULES" (`handleLevelJump`, gated on `game?.godMode`) that jumps `level += 5` and resets `destroyedThisLevel`, for reaching high-level kinds without playing through dozens of words first.

**Lives HUD is a battery bar, not hearts.** `page.tsx` computes `livesPct = (lives / LIVES_START) * 100` and a three-stop color (`var(--success)` >60%, `var(--warning)` >30%, else `var(--danger)`), rendered as `.battery`/`.battery-fill`/`.battery-nub` (globals.css) — a fixed-width outer shell with a width-and-color-animated inner fill. If `LIVES_START` changes, nothing else needs touching; the percentage calc is already relative.

## Lane placement & edge clipping

`spawnWord` assigns each word a `lane` (0..`LANES-1`) and a `baseX` via `laneToX`. Falling-word `<div>`s are centered on their `x` with `transform: translateX(-50%)` (`.breach-word` in `globals.css`), and `.breach-field` clips overflow — so a lane center too close to 0%/100% lets a wide word's near edge get cut off. `EDGE_MARGIN` (15, in `engine.ts`) reserves that much percentage on both sides; `laneToX` distributes lanes only within the remaining middle band, and `erratic`'s sine-wave drift is clamped to the same `[EDGE_MARGIN, 100 - EDGE_MARGIN]` range so jitter can't push a word back out past the margin. If `LANES` or the word bank's max length changes meaningfully, re-check this margin is still generous enough at the narrowest supported viewport.

## Pausing for the rules modal

The tick `useEffect` depends on `[showRules]` and returns early (no interval) while the modal is open — same gating pattern as the other games. The non-obvious part: on **resume**, every absolute timestamp already stored in `gameRef.current` (`now`, `lastSpawnAt`, each word's `spawnedAt`, each effect's `expiresAt`) gets shifted forward by exactly how long the modal was open, *before* the new interval starts. Skipping this would otherwise cause, on resume: a giant one-tick `deltaY` jump (huge `now - state.now`), an instant flood of overdue spawns (`now - lastSpawnAt` already past the spawn interval), erratic words jumping to a wild phase, and slow/boost effects expiring "early" relative to wall-clock play time. If you add another absolute-timestamp field to `GameState`/`FallingWord`/`SpeedEffect`, it needs the same `+= paused` treatment in this effect.

## Level-up

Tracked by `destroyedThisLevel` (incremented on every `'destroyed'` outcome in `submitInput`, including the second hit of a shield — *not* the first — and a cleared `chained` pair, which only increments once for both halves). At `WORDS_PER_LEVEL` (4 — deliberately low so the 15 kinds unlock at a reasonable pace; it was 8 originally), `level += 1` and the counter resets. Leveling up does three independent things, all keyed off the same `level` number: raises `fallSpeedForLevel`, shrinks `spawnIntervalForLevel`, and (via `KIND_UNLOCK_LEVEL`) widens the pool `pickKind` draws from. Speed and spawn-rate scaling cap out at `MAX_LEVEL_FOR_SCALING` (12) so the game stays winnable-feeling at high levels even though kind variety keeps unlocking all the way through level 15 and stays maxed after.

## If extending

- New kind: add to `WordKind`/`ALL_KINDS`, give it an unlock level in `KIND_UNLOCK_LEVEL`, a `KIND_STYLE` entry (accent color + rules hint — this also drives the rules-modal listing automatically), a weight in `engine.ts`'s `KIND_BASE_WEIGHT`, and decide which of the three mechanics it touches: display (`WordGlyphs`/`kindIcon`/`wordAccent` in `page.tsx`), required-input/matching (`requiredInput` in `constants.ts`, or a special case in `submitInput` like `caseSensitive`'s), or behavior-on-destroy/breach (`submitInput`/`tick` in `engine.ts`, like `spawner`/`shifting`). Most kinds only touch one of the three. If the new kind needs more than one `FallingWord` (like `chained`), branch out of `spawnWord` early the same way `chained` does, and remember every `Record<WordKind, ...>` map (`KIND_BASE_WEIGHT`, `scoreForWord`'s `multiplier`) needs an entry too or TS will fail the build. If it needs a brand-new per-word field (like `nextShiftAt`/`garbledText`), every other `FallingWord` literal across `engine.ts` (`spawnWord`, `spawnChainedPair`'s `makeHalf`, `spawnChildrenAt`) needs a default for it too, or TS will fail the build the same way.
- New word lengths/themes: extend `WORDS_SHORT`/`WORDS_MEDIUM`/`WORDS_LONG` (single words), `CHAIN_PAIRS` (two-word collocations — keep these as phrases that read naturally together), or `WORDS_CASE_SENSITIVE` (mixed-case literals, never upper-cased) — `poolForKind` already biases longer words at higher levels and separately filters pools for `skipLetter` (wants length ≥ 5) and `caseSensitive` (its own dedicated bank, bypassing the tier roll entirely); keep those filters in mind if the bank composition changes drastically.
- Tuning difficulty: nearly everything is a named constant in `constants.ts` (`SPAWN_INTERVAL_*`, `BASE_FALL_SPEED`/`SPEED_PER_LEVEL`, `SLOW_FACTOR`/`BOOST_WORD_MULTIPLIER`/`CHAIN_SCORE_MULTIPLIER`, `SHIFT_INTERVAL_MS`, `SPAWNER_CHILD_COUNT`, `ENCRYPTED_REVEAL_Y`, `WORDS_PER_LEVEL`, `LIVES_START`) — prefer adjusting those over touching `engine.ts`'s control flow.
